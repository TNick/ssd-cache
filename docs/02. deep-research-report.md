# Detailed implementation strategy for a Windows native NAS cache disk system

## Recommended implementation path

The central architectural conclusion is that a **pure user-mode application plus service is not enough** if you truly need to observe **both reads and writes** on files accessed through a mapped NAS drive. The user-mode directory notification APIs (`ReadDirectoryChangesW`, `ReadDirectoryChangesExW`) only report **changes inside directories** and explicitly point to change journals for volume changes; they do not provide reliable read-access monitoring. By contrast, Windows file-system **minifilter drivers** are specifically designed to monitor, filter, or modify file I/O operations, and MicrosoftŌĆÖs own MiniSpy sample is a minifilter that logs I/O activity from both kernel- and user-mode components. For this product, the lowest-risk design is therefore a **three-part system**: a small **kernel minifilter** for access observation, a **user-mode Windows service** for scheduling and copying, and a **tray application** for user control and mode switching. ŅłĆciteŅłéturn24search2Ņłéturn31search5Ņłéturn18search0Ņłü

A second key conclusion is that the **service must not rely on mapped drive letters** to reach the NAS. MicrosoftŌĆÖs service guidance is explicit: drive letters are **not global**, each logon session has its own drive mappings, and a service cannot access drive letters created in another logon session. Microsoft also says a service should use the **UNC path** instead of a mapped letter and should not manage mapped drive letters through `net use` or the Windows networking functions from service context. That matters here because your UI and ŌĆ£monitor/serveŌĆØ modes are drive-letter based, but the service must internally treat the source as a **share identity** such as `\\server\share`, not `N:`. The tray app may still manage the **interactive userŌĆÖs** mapped letter for presentation, but the service should store and operate on the **UNC root + relative path**. ŅłĆciteŅłéturn20view0Ņłéturn19search2Ņłü

Because of those two facts, the best v1 plan is this: keep the **business logic portable** in C++, keep the **Windows integration** in a few isolated modules, and implement the only Windows-specific hard partŌĆöthe access monitorŌĆöas a minifilter driver. For future Linux support, keep the portable boundary at the level of ŌĆ£activity events,ŌĆØ ŌĆ£copy scheduling,ŌĆØ ŌĆ£path identity,ŌĆØ and ŌĆ£mount presentation,ŌĆØ so that a later Linux backend can swap in `fanotify` or `inotify`-based monitoring behind the same internal interfaces. LinuxŌĆÖs file monitoring facilities support similar event-driven designs, but the Windows read-monitoring requirement is what forces the minifilter on Windows. ## Target architecture and module boundaries

The implementation should be split into **portable core code** and **Windows adapters**. The portable core should not know about Win32 handles, service status handles, device notifications, drive letters, or SMB redirector names. It should know only about source roots, relative paths, access events, configuration, pending copy jobs, cache inventory, and mode state. Windows-specific code should translate between Win32/Filter Manager concepts and those neutral domain objects. This design keeps the path open for Linux later and also prevents the service logic from becoming tangled with session-specific drive mapping rules. ## Monitoring pipeline and copy scheduling

The minifilter should register for at least `IRP_MJ_CREATE`, `IRP_MJ_READ`, `IRP_MJ_WRITE`, and `IRP_MJ_CLEANUP`. Registration and startup follow the standard Filter Manager sequence: `FltRegisterFilter` during `DriverEntry`, then `FltStartFiltering`. MicrosoftŌĆÖs MiniSpy sample and Filter Manager guidance are the right starting point. The minifilter should communicate upward to the service using a Filter Manager communication port, again following the MiniSpy pattern. ŅłĆciteŅłéturn18search8Ņłéturn18search10Ņłéturn18search0Ņłü

The service-side event model should be deliberately simple:

- A **read** should mark the file as ŌĆ£recently seenŌĆØ and schedule or reschedule a delayed copy.
- A **write** should not cause immediate copy; it should mark a handle as dirty.
- A **cleanup** on a dirty handle should be interpreted as ŌĆ£the writing phase for this handle has ended,ŌĆØ which is the correct place to trigger the copy debounce window.
- The **debounce clock** should be reset to `now + configured_delay` on every new access to the same relative path.

That design matches the Windows cleanup semantics. Cleanup occurs when the last handle for the file object is being closed and is where drivers release handle-specific state; it is the right point to treat a write handle as ŌĆ£closed enoughŌĆØ for post-write scheduling. To implement this safely, the minifilter should use a **stream-handle context** to remember, per opened handle, whether write I/O occurred. Filter Manager explicitly supports stream-handle contexts through `FltAllocateContext`, `FltSetStreamHandleContext`, `FltGetStreamHandleContext`, and `FltDeleteStreamHandleContext`, and Microsoft ships a dedicated CTX sample for this pattern. ŅłĆciteŅłéturn21search4Ņłéturn21search6Ņłéturn38search2Ņłéturn38search0Ņłéturn38search1Ņłéturn38search3Ņłéturn38search11Ņłü

The service must also suppress **self-generated I/O**, otherwise its own reads from the NAS during copy will recursively generate more access events. Filter Manager exposes the requestor information directly: the `FLT_CALLBACK_DATA` structure represents the I/O operation, and Filter Manager provides routines including `FltGetRequestorProcessId`, `FltGetRequestorProcess`, and requestor session helpers. The simplest production rule is: the service registers its PID with the driver on start, and the driver ignores events whose requestor PID matches the service process. You should also ignore minifilter-generated I/O where the callback data flags indicate reissued/generated I/O. ŅłĆciteŅłéturn37search0Ņłéturn37search3Ņłü

In user mode, implement the pending-copy scheduler as a **single writer thread plus a min-heap ordered by due time**. Do not create one timer per file. Each access event performs an UPSERT into SQLite and also updates an in-memory `next_due` entry. The worker thread sleeps until the nearest due job, wakes, re-checks the latest due time for that path, and skips stale heap entries if a newer access already pushed the deadline out. This pattern is deterministic, cheap, and easy to persist through restarts by rehydrating due jobs from the database when the service starts. The worker thread should enter **background processing mode** so the system lowers its resource scheduling priorities, and the actual file handles can also receive a low I/O priority hint. Microsoft explicitly documents that `THREAD_MODE_BACKGROUND_BEGIN` and `PROCESS_MODE_BACKGROUND_BEGIN` reduce background resource impact, and `FILE_IO_PRIORITY_HINT_INFO` can set file-handle I/O hints such as `IoPriorityHintVeryLow`. ŅłĆciteŅłéturn9search0Ņłéturn9search1Ņłéturn9search2Ņłéturn10search0Ņłéturn10search3Ņłü

For the actual copy, do **not** use `CopyFile2` as the primary path, because your design requires ŌĆ£copy and hash in one passŌĆØ so that the source file is not read twice. Instead, open the source with `CreateFileW`, stream it in chunks with `ReadFile`, feed the same chunks to `BCryptHashData`, and write them to the destination with `WriteFile`. That satisfies your ŌĆ£hash must be part of the copy operationŌĆØ requirement exactly. It also lets you throttle naturally by chunk size and small sleeps between iterations if you find background mode alone insufficient on slower machines. ŅłĆciteŅłéturn11search0Ņłéturn11search3Ņłéturn11search1Ņłéturn8search0Ņłéturn8search1Ņłéturn8search4Ņłü

Two important correctness details should be added even though your original plan did not call them out explicitly. First, the minifilter should also report **rename** and **delete** activity, because otherwise the cache and database will accumulate stale paths. Second, all path handling should be **Unicode-first** and long-path aware, because network shares and deep cache roots can exceed legacy `MAX_PATH` limits. Windows documents both the default legacy path limitation and the long-path opt-in model. ŅłĆciteŅłéturn33search0Ņłéturn33search1Ņłéturn33search12Ņłü

## Database, hashing, and overwrite policy

For storage, SQLite is a very good fit for this service because it is embeddable, very small, and the official amalgamation contains everything needed to embed SQLite into a native application. The simplest integration is to vendor the upstream `sqlite3.c` and `sqlite3.h` amalgamation into the tree instead of introducing a wrapper library dependency. SQLiteŌĆÖs own documentation describes the amalgamation as the complete embeddable form and provides the C/C++ API directly. ŅłĆciteŅłéturn29search1Ņłéturn29search0Ņłéturn7search0Ņłéturn7search9Ņłü

The database should use at least two tables in v1:

```sql

    relative_path TEXT PRIMARY KEY,
    last_seen_utc TEXT NOT NULL,
    last_write_close_utc TEXT,
    source_size_bytes INTEGER,
    cached_size_bytes INTEGER,
    cached_at_utc TEXT,
    hash_algo TEXT,
    hash_hex TEXT,
    hash_at_utc TEXT,
    cache_present INTEGER NOT NULL DEFAULT 0,
    last_error TEXT
);

CREATE TABLE pending_jobs (
    relative_path TEXT PRIMARY KEY,
    due_utc TEXT NOT NULL,
    reason TEXT NOT NULL,
    retries INTEGER NOT NULL DEFAULT 0,
    FOREIGN KEY(relative_path) REFERENCES files(relative_path)
);
```

`files.relative_path` should be relative to the **source share root**, not the presentation letter. When the driver supplies a remote normalized name, parse out the `\server\share` portion and persist only the remainder. That is the direct equivalent of your requested ŌĆ£path relative to the disk without the disk letter,ŌĆØ but expressed in a service-safe way that survives different logon sessions and future Linux backends. ŅłĆciteŅłéturn31search1Ņłéturn20view0Ņłü

SQLite operations should use prepared statements and UPSERT. The critical path is the access-event update, which should be a single UPSERT by `relative_path`, updating `last_seen_utc`, the most recent `source_size_bytes` when known, and the pending due time. SQLite supports UPSERT syntax directly through `ON CONFLICT ... DO UPDATE`. On startup, open the database with `sqlite3_open_v2`, set `PRAGMA journal_mode=WAL`, use a modest `busy_timeout`, and consider `synchronous=NORMAL` for a good durability/performance balance in a single-writer service. SQLiteŌĆÖs documentation explains WAL mode, the meaning of the `synchronous` settings in WAL mode, and the UPSERT syntax. ŅłĆciteŅłéturn7search3Ņłéturn7search10Ņłéturn7search1Ņłéturn7search2Ņłü

For hashing, default to **SHA-256** using CNG. Store `hash_algo='sha256'`, the lowercase hex digest, and `hash_at_utc`. The overwrite decision should **not** use the hash by default, exactly as you requested. The default rule should be:

1. If the cache file does not exist, copy.
2. If the path differs, treat as different file.
3. If sizes differ, overwrite.
4. If sizes match, skip by default.
5. If the user enabled ŌĆ£compare hash before overwrite,ŌĆØ compute and compare hash and warn in the UI that this increases I/O and CPU load.

That policy is practical because the hash is already available after a completed copy, but forcing a re-hash of the source for every same-size file would turn the copy scheduler into a verification engine and directly undermine your ŌĆ£slow pace, low impactŌĆØ goal. CNG supports SHA-2 algorithms natively, including SHA-256, and the streaming hash pattern is exactly what the CNG examples describe. ŅłĆciteŅłéturn8search0Ņłéturn8search15Ņłéturn8search17Ņłü

## Mode switching and operational behavior

Your three modes are sound, but they must be split between **interactive presentation logic** and **privileged storage logic** because Windows handles network drive letters and local volume mount points differently. In **disabled** mode, the tray app should stop asking the service to schedule new work and should leave the currently presented letters in a neutral state. In **monitor** mode, the tray appŌĆörunning in the interactive userŌĆÖs logon sessionŌĆöshould ensure that the network share is mapped to the selected source letter with `WNetAddConnection2` or `WNetAddConnection3`, while the privileged side ensures the cache volume is mounted to its own cache letter with `SetVolumeMountPoint`. In **serve** mode, the tray app should unmap the network drive letter in the user session using `WNetCancelConnection2`, and the privileged side should delete the cache diskŌĆÖs current mount point and reassign that volume to the sourceŌĆÖs presentation letter using `DeleteVolumeMountPoint` and `SetVolumeMountPoint`. ŅłĆciteŅłéturn6search5Ņłéturn6search11Ņłéturn19search17Ņłéturn6search1Ņłéturn6search4Ņłéturn6search6Ņłéturn20view0Ņłü

A subtle but important product limitation follows from MicrosoftŌĆÖs session model: because mapped network drives are **per logon session**, while local volume mount points are **system-wide**, v1 should be explicitly scoped to a **single interactive user** on a workstation. If multiple users are signed in simultaneously, they may each have different mappings for the same source letter, but the cache diskŌĆÖs local letter assignment is global. If you do not set that scope boundary, the product will produce surprising behavior in Fast User Switching or multi-session setups. ŅłĆciteŅłéturn20view0Ņłü

The tray app itself can remain lightweight and native. It should use `Shell_NotifyIcon` for the notification-area presence and a hidden message window for the context menu and `TaskbarCreated` handling. It should control service start/stop through the SCM APIs and listen for status transitions with `NotifyServiceStatusChange` rather than polling aggressively. Configuration changes should be written to a file in `ProgramData`, then the app should notify the service with `SERVICE_CONTROL_PARAMCHANGE` or a user-defined service control code to reload immediately. Microsoft documents both the tray notification model and the service status notification model. ŅłĆciteŅłéturn5search1Ņłéturn32search0Ņłéturn32search3Ņłéturn16search0Ņłéturn25search4Ņłü

MicrosoftŌĆÖs `ProgramData` guidance exists exactly for machine-wide application data that standard users must access without storing operational state under `Program Files`. Use `SHGetKnownFolderPath` to resolve the actual path rather than hardcoding it. ŅłĆciteŅłéturn25search0Ņłéturn25search4Ņłü

## Risks, constraints, and open questions

The biggest engineering risk is not the copier or the tray UI; it is the **driver requirement** introduced by the ŌĆ£monitor reads and writesŌĆØ requirement. If your team wants to avoid a kernel component, you should consciously narrow the requirement to ŌĆ£monitor creates/modifies/renames/deletesŌĆØ and then use a user-mode watcher such as `ReadDirectoryChangesW`. As soon as true read observation remains required, the minifilter should be treated as part of the product scope, build tooling, signing pipeline, and QA budget from day one. The most important operational constraints are these. First, **internal source identity must be UNC-based**, not drive-letter based, because services cannot safely depend on user-session mappings. Second, mode switching is split across **user-session network mappings** and **global volume mount points**, which makes multi-user support a separate project, not a free add-on. Third, the cache worker must suppress self-generated source reads or else it will feed back into the monitor path. Fourth, production rollout on unmanaged endpoints will require a real driver-signing and update strategy, not just an MSI. ŅłĆciteŅłéturn20view0Ņłéturn6search1Ņłéturn23search2Ņłéturn37search0Ņłü
